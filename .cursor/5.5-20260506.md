We have a problem in our @src/services/synapsd/ toy db, specifically in @src/services/synapsd/src/views/ContextTree.js  and @src/services/synapsd/src/views/DirectoryTree.js and the @src/services/synapsd/src/schemas/internal/layers/Canvas.js abstraction.
Our toy database uses 2 virtual tree types as abstractions on top of indexed user data:
- "context" tree, all tree nodes are linked to bitmaps, path /foo/bar/baz does a bitmap AND for layer(node) foo AND bar AND baz
- "directory" tree, tree nodes maped to bitmaps but follow a more familiar schemantics - each layer is a self-contained "folder"

Both trees support the same basic CRUD schemantics, context tree additionally support bitmap schemantics for merge/subtract layers.

Context tree nodes are unique, "reports" layer in /customer/reports and /invoicing/reports is the same layer - by design, placecment of a layer within the context tree filters down the data

We support a set of feature and filter bitmaps on top of this path based filtering and several layer types:
- standard "context" layer
- layer of type "canvas"
  - Canvas is a movable (within the tree) stored view, it contains a stored set of filters, features and metadata; user can create a canvas "Todays emails" which would filter down all data based on a set of filters and optionally store application UI/layout/widget data in the layer metadata. Such a canvas can be reused across the tree when needed or even shared/made public on the app level
  Canvas itself does not use its own name for the path filter, path /foo/bar/baz whehre baz is a canvas uses /foo/bar as a context filter in a context tree, in a directory tree the parent "folder"
- Later we want to support a layer of type workspace - mountpoint to a remote workspace tree
- maybe we should introduce a "directory" type layer for use in the directory tree

Now the problem:

1. Create canvas "Files" when context layer "files" exists — getLayerByName returns the existing layer

  2. Drop canvas "foo" onto a path that already has "foo" elsewhere — addChild/saveTree probably resolves to

  3. Create canvas "test1" while /foo/test1 exists — insertPath finds existing test1 by name, hits the
  "Upgrading leaf layer from context to canvas" branch (ContextTree.js:474-477), silently converts.
